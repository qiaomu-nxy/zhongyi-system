# 中医问诊管理系统 — 项目完整上下文

> 本文档记录了所有需求讨论和设计决策，供任何 agent 或开发者阅读参考。
> 最后更新：2026-03-20

---

## 一、项目概述

为中国大陆的中医馆（小团队/单个诊所）开发一套问诊管理系统。核心治疗方式为**中药和针灸**。

**核心需求：**
- 患者扫码填写基本信息和病症，避免每次重复填写
- 能查询是第几次就诊，有完整历史记录
- 患者可在线预约就诊时段（限未来7天内）
- 医师账号密码登录后台，查看所有患者信息，撰写病历和治疗方案
- 医师能直观分析患者治疗过程中的变化（ECharts图表）
- 老年患者/不会使用手机的患者，医师可手动代为创建就诊记录

## 二、版本规划

### V1（当前开发）— 单诊所单医师，快速验证
- 患者端：H5 移动网页（扫码用手机浏览器打开）
- 医师端：Web 响应式后台（电脑和手机都能用）
- 后端：Python FastAPI + SQLite
- 部署：Railway（三端分服务部署，用于验证环境）
- 语音输入：依赖手机键盘自带语音，不需额外开发
- 处方格式：纯文字输入
- 角色：保留 `admin` + `doctor`，但 V1 默认仅 1 位医师接诊

### V2（后续升级）— 单诊所多医师，正式运营
- 患者端切换为微信小程序（uni-app + Vue 3，需营业执照+微信审核）
- 后端迁移到云服务器（阿里云/腾讯云，约50-100元/月）
- 处方格式升级为结构化（每行：药名+克数+炮制方式）
- AI辅助：接入 DeepSeek API 做病历智能分析
- 升级为多医师排班、患者预约选医师、按医师维度的数据权限

### V1 → V2 升级路径
- 后端代码和数据库完全不变，直接搬迁
- 患者端从 H5 切换为 uni-app 小程序（API 接口一致，只改前端）
- 数据库文件（zhongyi.db）复制到新服务器，数据零丢失
- 迁移耗时约 30 分钟

## 四、UI 设计规范

参考 Google Material Design 3，整体风格轻盈通透，大面积留白。

### 配色方案
- **主色**：柔和薄荷绿/鼠尾草绿（Sage Green `#5DB391`），绿色只作为点缀
- **页面背景**：纯白 `#FFFFFF` + 极浅灰 `#FAFAFA`
- **卡片**：白底 + 柔和阴影（不用硬边框）
- **病症选中状态**：浅珊瑚/浅红色 `#EF9A9A`，与绿色形成对比，标明"病症区"
- **未选中状态**：白底 + 浅灰边框
- **成功/完成**：浅绿色
- **取消/错误**：柔和红色
- **文字**：深灰 `#333333`

### 设计原则
- 圆角 16px，Material 3 药丸形按钮
- 大量留白，呼吸感强
- 绿色用于：导航、操作按钮、标签、时间线等常规 UI 元素
- 浅红/珊瑚色用于：症状填写页的**已选**症状标签、人体图选中部位、严重程度滑块（与整体绿色调区分，标明"这里是填病症的"）
- 医师端响应式：桌面端完整布局，手机端自动调整为单列

### Logo
- 风格：现代中医风，融合中药（草药叶片）和针灸（银针）两个元素
- 颜色：薄荷绿主色
- 无文字，纯图标形式
- 参考设计稿：`assets/logo-herb-needle.png`

## 六、功能详细说明

### 6.1 医师端 — 账号登录
- 医师通过用户名 + 密码登录管理后台
- 密码使用 bcrypt 哈希存储，不明文保存
- 支持两种角色：`doctor` + `admin`
- 登录态通过 JWT token 维持，token 存于浏览器 localStorage
- 医师忘记密码由 `admin` 后台重置
- V1 默认仅 1 位 `doctor` 接诊，多医师协作放到 V2

### 6.2 患者端 — 手机号 + 姓名登录
- V1 使用“手机号 + 姓名”识别患者
- 首次：患者不存在 → 填写基本信息，创建档案
- 再次来：患者已存在 → 显示"欢迎回来，张三，这是您第N次就诊"，跳过基本信息
- V2 升级为微信 openid 自动识别，无需手动输入

### 6.3 患者端 — 结构化症状采集
**填写时机**：仅限就诊当天才能填写，确保信息反映当天实际状态。

采集方式（引导式，不让患者自由输入大段文字）：
1. **不适部位**：人体正面图 + 背面图点选，可多选，共 12 个区域
   - 头部、面部、颈部、肩部、胸部、腹部、腰部、背部、上肢、下肢、足部、全身
2. **症状勾选**：根据选中部位动态显示常见症状标签
   - 如选"头部" → 头痛/偏头痛/头晕/失眠/耳鸣/目眩...
   - 如选"上肢" → 手指发麻/手指冰凉/关节痛/乏力/水肿...
3. **严重程度**：滑块打分 1-10（滑块为浅珊瑚色）
4. **持续时间**：下拉选择（今天/几天/一周/一个月以上...）
5. **补充描述**：文本框（利用手机键盘自带语音输入）
6. 可添加多个症状

### 6.4 患者端 — 在线预约
- V1 无需选择医师，直接预约未来 7 天内的时段
- 根据诊所唯一医师排班自动显示可选时段（已约满/休息日不可选）
- 当天剩余时段在截止时间前也可预约
- 选择日期 → 选择时段 → 确认预约
- 可查看预约记录，**就诊前1天（24小时前）**可自行取消，当天不可取消
- 若需改期，只能先取消再重约
- 患者取消后，时段立即重新开放

### 6.5 医师端 — 今日就诊概览（Dashboard）
- 统计卡片：今日预约数、待接诊数、已完成数、今日新患者/复诊患者
- 今日列表按实际签到/创建时间排序
- 待接诊列表：姓名、性别、年龄、就诊次数、预约时间、主要症状、"开始接诊"按钮
- 已提交症状的患者有状态标记，未提交的显示"待填写"
- 已完成列表：同上，操作列为"查看详情"

### 6.6 医师端 — 手动创建就诊（解决老年患者问题）
- 医师可搜索已有患者 → 手动创建当天就诊记录
- 若患者未在系统中 → 医师可代填基本信息，创建新患者档案
- 手动创建的就诊记录标注 `created_by = 医师手动创建`

### 6.7 医师端 — 患者详情
- 顶部：基本信息卡片（姓名/性别/年龄/手机号/首诊日期/总就诊次数）
- 既往史：既往病史标签、过敏史、外部就诊史（机构+治疗方式+效果+副作用）
- V1 仅 1 位接诊医师，可直接查看本诊所患者的全部历史
- Tab 切换：
  - **就诊记录**：时间轴，每条可展开，显示症状数据 + 病历全文
  - **数据分析**：ECharts 图表（见 6.9）
  - **检验指标**：历次检验数据表格

### 6.8 医师端 — 撰写病历
左右分栏布局（手机端改为上下布局）：

左栏（60%）病历表单：
- 治疗方式：中药 / 针灸 / 推拿 / 艾灸 / 其他（根据选择动态显示对应字段）
  - 选"中药" → 显示处方文本框（V1纯文字）
  - 选"针灸" → 显示穴位标签输入框（如：合谷 × 足三里 × 百会 ×）
- 舌诊、脉诊
- 体征观察（面色/体型/痰核等）
- 诊断
- 辨证思路/治疗策略
- 治疗方案、备注
- 保存病历后自动完成本次就诊与关联预约
- 病历提交后普通医师不可修改，只有 `admin` 可修改

右栏（40%）患者本次填报信息（只读）：
- 已选症状列表（部位 + 症状名 + 严重程度条 + 持续时间）
- 补充描述文字
- 历史症状趋势迷你折线图（最近5次）

### 6.9 医师端 — 治疗变化分析（ECharts）
- **症状评分折线图**：X轴=就诊次数，Y轴=严重程度，每个症状一条线，直观看治疗效果
- **症状雷达图**：首诊 vs 当前，多维对比，看整体改善幅度
- **治疗时间轴**：每次就诊日期 + 关键诊断摘要，完整历程一目了然
- **检验指标趋势**：血糖/血压等数值变化柱状图，带参考值线（如血糖正常值6.1）

### 6.10 医师端 — 预约管理
- **预约日历**：周视图，每个时段的颜色状态：绿色=已预约、白色=可预约、灰色=休息、红色=已取消
- 点击已预约格子 → 弹出患者信息 + 查看详情 / 取消预约
- **时段设置**：按诊所唯一医师设置每周固定排班（星期几上班 + 上下午时段 + 时段间隔30/45/60分钟）
- **临时调整**：某天设为休息日或修改时段；若与已有预约冲突，系统自动取消冲突预约
- **预约规则**：可提前预约天数（默认7天）；患者最晚取消时间：就诊前1天（当天不可取消）；医师可随时取消
- **爽约规则**：患者未到诊时，由医师在当日营业结束后标记为爽约；V1 先记录次数，不限制预约
- **通知机制**：医师主动刷新页面查看新提交（小诊所流量小，足够用）
- **V2扩展**：升级为多医师独立排班、患者预约选医师

### 6.11 医师端 — 数据备份导出
- 一键导出全部数据为 Excel（patients + visits + medical_records）
- 下载数据库备份文件（zhongyi.db 单文件，复制即备份）
- 按单个患者导出完整病历档案

## 八、核心使用流程

### 患者端流程
```
扫码打开网页
    ↓
输入手机号 + 姓名
    ↓
新患者 → 填写基本信息（姓名/性别/出生日期/既往史）
    ↓
已有患者 → 自动识别，显示"欢迎回来，张三，第N次就诊"
    ↓
预约就诊（直接选未来7天内的可用时段）
    ↓
就诊当天到店签到
    ↓
就诊当天扫码 → 填写本次症状
（点选12部位人体图 → 勾选症状标签 → 打严重程度分 → 补充描述）
    ↓
提交等待医师接诊
    ↓
可查看历史摘要、预约记录和简短医嘱
```

### 医师端流程
```
账号密码登录后台
    ↓
首页看今日就诊列表
（今日预约患者 + 症状提交状态标记，按签到/创建时间排序）
    ↓
┌─ 常规接诊 ────────────────────┐
│ 点击患者 → 查看症状数据         │
│ 撰写病历（舌诊/脉诊/体征/诊断/  │
│ 辨证思路/处方or穴位/治疗方案）  │
│ 保存病历 → 自动完成接诊         │
└────────────────────────────────┘
    ↓
┌─ 老年/未扫码患者 ──────────────┐
│ 手动搜索患者 or 创建新患者       │
│ 医师代填信息 → 创建就诊记录      │
│ 撰写病历（同上）                 │
└────────────────────────────────┘
    ↓
患者详情页 → 数据分析Tab
（症状折线图/雷达图/时间轴/检验指标趋势）
```

## 九、部署方案

### V1 — Railway 验证环境
- 注册 GitHub 账号（免费）+ Railway 账号（验证期可用）
- 代码推到 GitHub，Railway 关联仓库自动部署
- 后端、患者端、医师端分别部署为 3 个 Railway 服务
- 获得公网域名（`*.up.railway.app`）
- SQLite 仍是单文件数据库，适合验证阶段和导出备份
- **限制**：中国大陆访问稳定性一般，适合验证，不建议长期正式运营

### V2 — 云服务器
- 阿里云/腾讯云最低配（约50-100元/月，24小时在线）
- 需要域名（约60元/年）+ HTTPS（Let's Encrypt 免费证书）
- HTTPS 为微信小程序必须项

### 迁移步骤（V1→V2，约30分钟）
1. 买云服务器，安装 Python
2. 复制代码仓库
3. 复制 `zhongyi.db` 数据库文件
4. 启动服务
5. 修改前端的 API baseURL 配置（一行配置，重新构建）

## 十、关键设计决策记录

| 决策点 | 结论 | 原因 |
|--------|------|------|
| 平台形态 | V1 H5 + V2 微信小程序 | 先零成本验证，再正式上线 |
| 医师模式 | V1 单医师 / V2 多医师 | 先快速落地，再扩展协作 |
| 后端语言 | Python FastAPI | 轻量、自带API文档、AI友好 |
| 数据库 | SQLite | 单文件、免安装、小诊所够用、备份方便 |
| 患者识别 | V1手机号 + 姓名 / V2微信openid | H5无法获取微信openid，且不接短信更省成本 |
| 医师登录 | 账号密码 + JWT | 保护医疗数据安全 |
| 医师权限 | V1 保留 `admin` + `doctor`，但仅 1 位 `doctor` 接诊 | 先保留角色结构，减少 V1 协作复杂度 |
| 多医师协作 | 患者预约选医师、按医师隔离数据放到 V2 | V1 先做单医师版本，减少开发范围 |
| 症状采集 | 结构化表单（非自由文本） | 减少无用信息，数据可直接用于图表 |
| 身体部位 | 12个区域，可多选 | 覆盖全身，粒度适中 |
| 症状填写时机 | 仅就诊当天 | 确保信息反映当天实际状态 |
| 老年患者 | 医师端手动代填 | 覆盖不会使用手机的患者 |
| 语音输入 | 依赖手机键盘自带语音 | 无额外开发成本，覆盖99%场景 |
| 预约规则 | 限7天内预约；患者可提前1天取消；医师随时可取消 | 给诊所足够的准备时间，当天不允许临时放鸽子 |
| 改预约规则 | 只支持取消后重约 | 简化状态设计，减少前后端复杂度 |
| 排班冲突 | 修改排班时自动取消冲突预约 | 便于现场快速调整 |
| 签到机制 | 到店先签到，再进入待接诊 | 让现场流程和队列更清晰 |
| 爽约处理 | 当日营业结束后由医师标记爽约，V1只记录不限制 | 先记录行为，避免早期过度限制 |
| 病历修改 | 提交后只有 `admin` 可改 | 降低医疗记录被随意篡改的风险 |
| 患者可见范围 | 只看历史摘要和简短医嘱，不看完整病历 | 保护医师专业记录 |
| 同日就诊 | 同一患者同一天只保留一条就诊记录 | 简化统计和流程 |
| 统计口径 | 新患者=第一次来诊所；复诊患者=以前来过至少一次 | 与“全诊所累计第N次”保持一致 |
| 处方格式 | V1纯文字 / V2结构化 | 先快速上线，后续优化 |
| 通知机制 | 医师主动刷新页面 | 小诊所流量小，不需要push通知 |
| 部署 | V1 Railway验证 / V2 云服务器 | 先验证流程，再迁移国内正式环境 |
| 数据备份 | 医师端一键导出Excel / 下载db文件 | 防止数据丢失 |
| UI风格 | Material Design 3 + 薄荷绿 | 轻盈现代，不压眼 |
| 病症区配色 | 浅珊瑚红选中 + 绿色背景 | 视觉区分病症填写区和操作区 |
| API版本 | /api/v1/ 统一前缀 | 方便未来版本共存 |
| 症状配置 | 后端JSON文件提供，前端不硬编码 | 方便后续增减症状类别 |

## 十一、真实病历参考案例

以下案例用于验证数据模型的完整性：

> **某男，44岁，素患糖尿病**，自觉双手指尖冒凉气，握方向盘时都发冷，右侧偏头痛，右手拇、食指发麻，畏寒。之前在一用药很刚的医生处看了2年，该医生为滋阴派，用药量大，近半年患者觉得身体畏寒、颜面发黑。
>
> 治疗采用"描针"（柔治）。首诊后偏头痛减轻，手指似有温热感，手麻减轻；二诊后双手觉温热，无头痛，手不麻；4诊后血糖7.0；6诊后肚子小了，颜面转红，不黑；8诊后诸症消失，痰核不见了。

该案例验证了以下字段的必要性：

| 表 | 字段 | 记录内容 |
|----|------|---------|
| patient_history | hospital_or_doctor, treatment_received, effect_note | 滋阴派医生2年治疗，副作用：畏寒、颜面发黑 |
| symptom_records | body_part=上肢, symptom_name=手指冰凉, severity=8 | 双手指尖冒凉气 |
| symptom_records | body_part=头部, symptom_name=偏头痛, severity=7 | 右侧偏头痛 |
| symptom_records | body_part=全身, symptom_name=畏寒, severity=7 | 全身畏寒 |
| medical_records | treatment_type=针灸, acupoints=描针 | 治疗方式 |
| medical_records | treatment_strategy | 柔治（对应之前滋阴派的"刚治"） |
| medical_records | physical_signs | 颜面发黑→转红，痰核，肚子变小 |
| lab_results | indicator_name=血糖, indicator_value=7.0 | 第4诊血糖记录 |
| visits（多条） | visit_number 1-8 | 8次就诊的完整历程 |
